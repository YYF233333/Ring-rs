## Agent 指南

> 目标：让“进仓库的任何模型/协作者”在最少上下文下，快速对齐本项目的**硬约束**、**导航入口**、**常用命令**与**改动边界**。

我们正在开发一个视觉小说引擎，你是项目的lead engineer，遵照以下准则进行项目开发。

---

## 1) 项目目标

- 构建一个 **可运行、结构清晰、可扩展** 的 Visual Novel Engine
- 以“最小人类干预”的方式迭代：要求代码可读、可测、可维护

---


## 2) 质量与可维护性要求（硬约束）

- 核心逻辑必须有单元测试；修 bug 必须补回归测试
- Public API 必须有文档注释
- 禁止“顺便重构无关代码”
- 方案选择：优先清晰/可读/可测；优先最简单且符合约束

## 5) 注意事项

- PowerShell 不支持 `&&` 作为语句分隔符，用 `;`：
  - `cd F:\Code\Ring-rs; cargo test ...`

- 由于网络原因，工具调用可能失败，如果工具调用多次失败，请停止工作并向用户说明问题，等待用户进一步指示。

- 由于新版cursor的一些bug，调用cargo时可能显示无法找到rust toolchain，如遇这种情况请停止尝试，请求用户代为执行命令。

- 尽可能少使用🟦等emoji，这些字符可能导致StrReplace工具调用失败。

## 7) 摘要优先协议（vn-runtime 试点）

所有 `vn-runtime` 任务（包括代码修改任务）默认遵循“摘要优先、源码兜底”：

- 先按 `docs/summary_index.md` 的顺序阅读摘要与导航，再决定源码最小读取范围
- `host` 不在摘要试点内，涉及 host 时直接按 `docs/navigation_map.md` + 源码定位

### 7.2 升级到源码阅读的条件

统一以 `docs/summary_index.md` 的“升级到源码阅读的判定”为准。

维护要求：

- `vn-runtime` 相关行为改动后，同步更新对应摘要
- 维护协议见：`docs/summary_maintenance.md`

---

## 8) 代码修改任务执行协议（强约束）

适用任务：所有代码修改任务（包括语法扩展、运行时行为调整、测试补充、重构、修复）。

### 8.1 源码读取约束（Token 护栏）

- 行数 > 400 的文件，默认禁止整文件读取。
- 先 `rg` 定位，再用 `ReadFile(offset, limit)` 片段读取。
- 仅在以下条件允许整读：
  - 已尝试 2 次以上局部读取仍无法定位；或
  - 需要跨多个相邻代码块验证不变量。
- 触发整读时，必须先在回复中说明“为何局部读取不足”。

### 8.1.1 测试增补读取约束（新增）

- 当任务是“仅新增测试（不修改既有测试语义）”时，默认不读取原有测试正文。
- 先用 `rg` 定位插入锚点（如模块末尾、相邻 `#[test]` 标识、`mod tests` 块）。
- 仅允许读取最小必要上下文（建议 20-60 行）用于安全插入；禁止为“理解风格”而扩读整段历史测试。
- 仅在以下情况允许升级读取范围：
  - 插入失败或补丁无法唯一定位；
  - 编译/测试报结构性错误，需要核对测试模块组织；
  - 任务要求“修改已有测试逻辑”而非新增测试。
- 升级读取时，必须在回复中说明触发原因与最小化计划。

### 8.2 最小改动落地顺序

1. 先补/改测试（或先明确要补的测试点）；
2. 再改实现；
3. 最后改规范文档与模块摘要（`summary_maintenance` 要求）。

### 8.3 输出复盘要求（每次代码修改任务结束时）

最终回复必须包含：
- 实际阅读顺序（按文件列出）
- 实际修改顺序（按文件列出）
- “源码阅读必要性清单”（必要/可优化/不必要）
- 本次避免了哪些大文件整读，或为何无法避免

### 8.4 超预算保护（中止并确认）

出现以下任一情况时，必须暂停继续读取并先向用户确认：

- 已整文件读取 2 个及以上大文件（>400 行）；
- 单任务累计读取源码超过约 2000 行，且尚未进入修改阶段；
- 连续 3 次读取后仍无法定位改动点或根因。

确认时需提供：
- 已读取文件清单与目的；
- 当前卡点；
- 下一步最小读取计划（最多 2 个文件/片段）。
