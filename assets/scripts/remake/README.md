## 3. Ring vNext 脚本规范（开发内部版）

目标：为迁移与引擎实现提供统一语义接口；优先优化内容生产的人体工学，由引擎吸收复杂度。

### 3.1 基础书写规则

- 一行只表达一件事，避免复合技术语句。
- 对白统一写法：`角色名："文本"`。
- 叙事文本统一写法：`旁白："文本"`。
- 优先使用语义化命令，不暴露底层 ATL/矩阵/并行动画细节。
- 同等效果下，优先减少内容侧心智负担，而不是追求“单条语句实现简单”。
- 允许命令在引擎内部维护必要 hidden state / side effect，但必须可预测、可复现、可调试。

### 3.2 当前采用的高层命令

- `changeScene <img src="..."/> with Effect(...)`：换场。
- `changeBG <img src="..."/> with Effect(...)`：同场景内背景/CG切换。
- `<audio src="..."></audio> loop`、`stopBGM`：BGM 控制。
- `wait N`：定时停顿（秒）。
- `show <img src="..."/> as 角色 at 位置 with 效果`：角色调度与差分切换统一入口（入场/退场/站位/移动/差分衔接）。
- `hide 角色 with 效果`：角色退场。
- `sceneEffect ...`：镜头语言抽象（摇镜、模糊、推拉、遮罩等）。
- `titleCard ...`、`cutscene ...`：字卡与视频过场语义。

说明：
- 现有历史翻译稿中仍存在 `showStyle` / `stageActor` 记法；后续按 RFC 统一收敛到 `show`。

### 3.3 迁移过程约束

- 不做 Ren'Py 语法逐项映射。
- 先保证剧情时序、情绪节奏和演出意图，再落技术细节。
- 引擎暂未支持的能力，允许以“设想语义命令”先落稿，并在本文件记录后续实现计划。
- `show` 可以承载复杂语义，但必须遵守“行为确定 + 规则可文档化 + 运行可观测”三项约束，避免演化为意面逻辑。

## 4. 未实现特性与实现计划

以下是迁移中识别出的核心缺口，按落地优先级组织为 P0/P1/P2。

### 4.1 P0（先保证完整可玩 + 关键观感）

1) 跨文件脚本调度
- `callScript` / `returnFromScript`（或等价 call stack）
- 多脚本 label 全局索引与跨文件跳转

2) 演出基础闭环
- 统一转场描述（覆盖 dissolve/fade/black/rule 关键类型）
- `scene/show/hide` 的资源 alias 映射（兼容重制期命名）

3) 文本节奏与窗口控制
- `pause`（点击等待）与 `wait`（定时等待）并存规则
- `extend`、`window hide/auto` 等节奏能力
- `{w}` / `{nw}` / `{cps}` 等高频标签的语义化替代或兼容

4) 持久化与门控
- `persistent.*` 语义映射
- `full_restart` 等价流程语义
- 首通后章节门控（如 `complete_summer`）

### 4.2 P1（增强“像原作”的主观体验）

1) 高级镜头语言
- `camera` 块（zoom/blur/matrixcolor/parallel/ease/linear）语义承接
- 复合转场（如 `tran_*` 家族）统一映射

2) 音频高级控制
- 分通道控制（music/sound/voice）
- 淡入淡出、片段播放、运行时动态音量
- `bgmDuck` / `bgmUnduck` 的正式实现

3) 资源动态定义
- `layeredimage` / `image = Text(...)` 等运行时资源定义能力

### 4.3 P2（长期可维护与生产效率）

1) 过场媒体能力完善
- 视频播放、跳过、回流与状态一致性

2) 复刻检查与回归机制
- 演出片段漏映射检查
- 资源缺失与节奏标签丢失检查
- 章节流程与门控状态检查

3) 内容生产规范沉淀
- 形成稳定的开发内部脚本模板与章节切分规范

## 5. Ren'Py 与 Ring 设计目标对比

### 5.1 Ren'Py 侧重

- 高可客制化：脚本层可直接嵌入大量实现细节（ATL/Python/transform）。
- 表达能力强但技术门槛高，内容层常被迫理解底层机制。

### 5.2 Ring 重制目标

- 人体工学优先：脚本主要描述叙事意图与演出意图，尽量减少内容侧认知负荷。
- 引擎兜底：复杂视觉/音频/状态机细节由 runtime/host 承担。
- 可维护优先：避免脚本层“技术债直写”导致长期不可控。

## 6. 脚本转换中的刻意取舍（已执行）

1) 放弃 1:1 语法复刻，保留体验语义
- 未直接搬运 ATL、`camera`、复杂 `tran_*` 细节。
- 统一改写为 `sceneEffect`、高层 `show/hide` 等语义命令。

2) 放弃“脚本即代码”，转向“脚本即意图”
- 尽量不在剧本中保留 Python 逻辑写法。
- 用更自然的舞台调度语义替换技术脚本片段。

3) 放弃局部最优效果，换取全局可读与可迭代
- 先保证剧情时序、情绪曲线、关键镜头节点正确。
- 允许短期内部分转场效果“近似但不等式复刻”，后续由引擎能力补齐。

4) 放弃分散文档，收敛为单一事实源
- 本文件作为迁移现状、规范、路线图的唯一维护入口，降低维护成本与信息漂移风险。
