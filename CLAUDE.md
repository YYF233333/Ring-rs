## Agent 指南

> 目标：让"进仓库的任何模型/协作者"在最少上下文下，快速对齐本项目的**硬约束**、**导航入口**、**常用命令**与**改动边界**。

我们正在开发一个视觉小说引擎，你是项目的lead engineer，遵照以下准则进行项目开发。

---

## 1) 项目目标

- 构建一个 **可运行、结构清晰、可扩展** 的 Visual Novel Engine
- 以"最小人类干预"的方式迭代：要求代码可读、可测、可维护

---

## 2) 质量与可维护性要求（硬约束）

- 核心逻辑必须有单元测试；修 bug 必须补回归测试
- Public API 必须有文档注释
- 禁止"顺便重构无关代码"
- 方案选择：优先清晰/可读/可测；优先最简单且符合约束

---

## 5) 摘要优先协议

所有任务（包括代码修改任务）默认遵循"摘要优先、源码兜底"：

- 先按 `docs/summary_index.md` 的顺序阅读摘要与导航，再决定源码最小读取范围

摘要维护要求见：`docs/summary_maintenance.md`

---

## 6) 代码修改任务执行协议

适用任务：所有代码修改任务（包括语法扩展、运行时行为调整、测试补充、重构、修复）。

### 6.1 源码读取约束（Token 护栏）

- 行数 > 400 的文件，默认禁止整文件读取。
- 先 `rg` 定位，再用 `ReadFile(offset, limit)` 片段读取。
- 仅在以下条件允许整读：
  - 已尝试 2 次以上局部读取仍无法定位，或需要跨多个相邻代码块验证不变量。
  - 用户显式授权。

### 6.1.1 测试增补读取约束

- 当任务是"仅新增测试（不修改既有测试语义）"时，默认不读取原有测试正文；直接用 `rg` 定位插入锚点（如模块末尾、相邻 `#[test]` 标识、`mod tests` 块），读取最小必要上下文（建议 20-60 行）用于安全插入；禁止为"理解风格"而扩读整段历史测试。
- 仅在以下情况允许升级读取范围：
  - 插入失败或补丁无法唯一定位；
  - 编译/测试报结构性错误，需要核对测试模块组织；
  - 任务要求"修改已有测试逻辑"而非新增测试。

### 6.2 输出复盘要求（每次代码修改任务结束时）

若读写了超过10个文件，最终回复必须包含：

- 实际阅读、修改顺序（按文件列出）
- "源码阅读必要性清单"（必要/可优化/不必要）
- 如果整读了大文件，说明为何无法避免

### 6.3 超预算保护（中止并确认）

出现以下任一情况时，必须暂停继续读取并先向用户确认：

- 已整文件读取 2 个及以上大文件（>400 行）；
- 单任务累计读取源码超过约 2000 行，且尚未进入修改阶段；
- 连续 3 次读取后仍无法定位改动点或根因。

确认时需提供：

- 已读取文件清单与目的；
- 当前卡点；
- 下一步最小读取计划。

---

## 7) RFC 驱动计划流程

项目计划与重大方案默认采用 RFC 流程。

适用范围（例）：

- 脚本语法/语义变更（如命令收敛、解析规则调整）
- Runtime 与 Host 协议变更（`Command`、状态机、等待语义）
- 跨模块改造（影响 `vn-runtime`、`host`、工具链任两者及以上）

执行要求：

- RFC 文件放在 `RFCs/`，命名建议：`rfc-<topic>.md`
- 至少包含：背景、目标/非目标、提案、风险、迁移计划、验收标准（DoD）
- 实施前先对齐 RFC；实现中若偏离 RFC，需先更新 RFC 再改代码
- 与 RFC 对应的实现完成后，及时同步相关文档（语法规范/摘要/导航）
