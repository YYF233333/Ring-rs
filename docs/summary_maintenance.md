# 摘要维护协议（vn-runtime + host）

## 目标

确保摘要可持续可用，避免“摘要过期反而误导”。

## 更新触发条件

满足任一条件即触发更新：

- `vn-runtime` 或 `host` 公共 API 变化（类型、函数、字段语义）。
- `vn-runtime` 或 `host` 任一已建摘要模块行为变更。
- 新增或删除脚本语法节点、命令类型、等待状态、诊断规则。
- 修复 bug 且行为回归测试有新增/调整。

## 更新范围规则

- 改动落在哪个子模块，就至少更新该子模块摘要。
- `vn-runtime` 跨模块改动需同步更新 `docs/module_summaries/vn-runtime.md` 总览。
- `host` 跨模块改动需同步更新 `docs/module_summaries/host.md` 总览。
- 如果阅读顺序或使用协议变化，必须同步更新 `CLAUDE.md`。

## 新鲜度与状态

- 每份摘要必须包含 `LastVerified`。
- 超过 30 天未校验，标记 `stale`。
- 标记为 `stale` 的摘要不可作为最终依据，必须回源核对。

## 质量检查清单

- 摘要中的文件路径仍存在。
- 描述与当前实现一致（至少抽查 2-4 个关键入口）。
- `WhenToReadSource` 判定清晰可执行。
- RelatedDocs 链接可跳转。
- `## Owner` 填写完成本次 `LastVerified` 时，校验者所使用的底层模型名称，如无法获知则填写**自认所对应的模型名称**（如实填写，不虚构）。
- 禁止使用「AI」「助手」等空泛词；若确实无法确定身份，写 `未记录` 并在后续校验中补全。
- 不使用仓库名、团队名等项目署名代替上述名称。

## 建议节奏

- 每周一次轻量抽查（5-10 分钟）。
- 大改后在同一 PR/提交中同步更新摘要，避免滞后。
