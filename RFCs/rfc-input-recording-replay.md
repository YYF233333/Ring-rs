# RFC: 输入录制与回放系统

## 元信息

- 编号：RFC-016
- 状态：Proposed
- 作者：Ring-rs 开发组
- 日期：2026-03-18
- 相关范围：`host/src/input`、`host/src/app`、`host/src/config`、`vn-runtime` 输入消费边界
- 前置：无

---

## 1. 背景

当前 Ring-rs 在 GUI 集成类问题（如 `egui` 覆盖层、点击命中、等待状态切换）上的调试流程依赖人工重复操作窗口。对 AI 辅助开发而言，模型无法直接操作本地窗口，导致以下问题：

- 同一问题每次改动后都需要用户手工重现，反馈回路慢；
- 复现步骤容易因操作节奏、点击位置、窗口状态不同而漂移；
- 当问题与时序相关（按键/点击间隔、等待态切换时机）时，口述步骤难以精确传递。

现有输入流大致为：

1. `InputManager::process_event(WindowEvent)` 接收 `winit` 事件并更新内部键鼠状态；
2. `InputManager::begin_frame(dt)` 推进计时；
3. `InputManager::update(waiting, dt)` 基于 `WaitingReason` 生成 `RuntimeInput`；
4. `InputManager::end_frame()` 清理逐帧状态。

该结构已经形成清晰输入边界，具备植入录制与回放能力的条件。目标是让用户“录制一次，复用多次”，使后续调试能够在无人工交互下复现输入路径，并通过日志/状态验证修复效果。

---

## 2. 目标与非目标

### 2.1 目标

- 提供一个可落地的输入录制与回放机制，支持将一次真实操作保存为文件并重复执行。
- 录制数据采用语义化事件而非原始平台事件，确保跨平台/跨版本稳定性与可读性。
- 在 `DebugConfig` 中提供配置开关，允许通过配置或命令行启用录制/回放。
- 回放期间可替换真实输入源，实现“无需人工操作窗口”的调试流程。
- 设计中明确时序与确定性策略，保证脚本逻辑层可重复。

### 2.2 非目标

- 不追求像素级渲染帧一致性（GPU、驱动、刷新率差异导致不可避免偏差）。
- 不覆盖音频播放时间轴的严格一致性（仅保证输入驱动逻辑可重放）。
- 不在本 RFC 中引入完整宏系统（如循环回放、条件分支、脚本化断言 DSL）。
- 不修改 `vn-runtime` 业务语义，仅在 host 输入源层做注入与替换。
- 不在首期支持跨分辨率自动重映射到“视觉目标元素”（仅提供基础坐标归一化策略）。

## 3. 方案设计

### 3.1 录制层级决策

候选方案：

- 方案 A：录制原始 `WindowEvent`
  - 优点：最忠实底层输入；
  - 缺点：数据体积大、平台细节重、与 `winit` 事件结构强耦合，文件可读性差。
- 方案 B：仅录制 `RuntimeInput`
  - 优点：最精简，直接贴近 `vn-runtime`；
  - 缺点：丢失鼠标位置与部分时序细节，难覆盖 UI 命中/悬停类问题。
- 推荐方案（本 RFC 采用）：录制 `InputManager` 语义中间层事件
  - 在 `process_event` 将原始事件映射为可重放的 `InputEvent`（如按键按下/释放、鼠标按下/释放、点击、移动、滚轮）；
  - 事件携带必要上下文（时间戳、坐标、按键、按钮），不绑定 `winit` 原始字段；
  - 在回放时再注入到 `InputManager` 同层逻辑，保持与真实路径一致。

该方案在“足够保真”和“可维护/可读”之间取得平衡，适用于 GUI 集成调试场景。

### 3.2 数据模型与文件格式

文件格式：JSON Lines（`.jsonl`），每行一条事件，UTF-8。

示例：

```json
{"t_ms": 1234, "event": {"type": "MouseClick", "x": 100, "y": 200, "button": "Left"}}
{"t_ms": 2345, "event": {"type": "KeyPress", "key": "Space"}}
```

建议结构：

- `t_ms: u64`：相对录制起点的毫秒时间戳；
- `event: InputEvent`：语义事件载荷；
- 可选元数据头（首行特殊记录或单独 metadata 文件）：
  - 录制时窗口逻辑尺寸（`logical_width`/`logical_height`）；
  - DPI scale；
  - 版本号（用于未来向后兼容解析）。

`InputEvent` 建议首期包含：

- `KeyPress { key }`
- `KeyRelease { key }`
- `MouseMove { x, y }`
- `MousePress { button, x, y }`
- `MouseRelease { button, x, y }`
- `MouseWheel { delta_x, delta_y }`
- `TextInput { text }`（可选，若后续文本输入场景需要）

说明：`MouseClick` 可由按下/抬起组合推导；如当前逻辑对点击有专门处理，也可保留独立事件以降低回放重建复杂度。

### 3.3 模块划分与职责

新增模块：`host/src/input/recording.rs`

- `InputRecorder`
  - 接收 `InputEvent` + 当前相对时间；
  - 缓存事件队列；
  - 在 `stop` 或 `drop` 时写出 `.jsonl`；
  - 写盘失败通过日志上报，不影响主流程继续运行。
- `InputReplayer`
  - 启动时读取并解析 `.jsonl`；
  - 维护“待注入事件游标”；
  - 按当前回放时间吐出到期事件；
  - 解析错误返回结构化错误（不使用字符串扁平化）。
- `InputManager` 扩展字段（可选）：
  - `recorder: Option<InputRecorder>`
  - `replayer: Option<InputReplayer>`
  - `replay_mode: bool`（或由 `replayer.is_some()` 推导）

`AppState` / 启动流程负责根据配置构造 recorder/replayer 并注入 `InputManager`。

### 3.4 帧循环集成

建议将回放注入点放在与真实事件同层级处理入口，确保状态机路径一致：

1. `process_event(WindowEvent)`：仅处理真实系统事件；
2. `begin_frame(dt)`：推进时间；
3. 若 `replayer` 存在，则在本帧取出 `t_ms <= elapsed_ms` 的事件，调用统一内部入口（例如 `process_input_event(InputEvent)`）；
4. `update(waiting, dt)`：基于内部状态生成 `RuntimeInput`；
5. `end_frame()`：清理逐帧状态。

关键点：真实输入与回放输入统一经过同一内部处理函数，避免逻辑分叉。

### 3.5 录制与回放控制

控制方式：

- 配置：
  - `debug.record_input: Option<String>`（设置则开启录制并写入路径）
  - `debug.replay_input: Option<String>`（设置则开启回放并读取路径）
- 运行时热键（建议 `F8`）：
  - 若未录制则开始录制；
  - 若正在录制则停止并写盘；
  - 回放模式下按热键可选择“中止回放并切回真实输入”（可选策略）。

冲突策略：

- 同时配置录制与回放时，默认优先回放并禁用录制；
- 或者启动时报错并要求用户二选一（二者可通过配置校验明确）。

### 3.6 回放时真实输入处理策略

默认策略：回放期间忽略真实输入（除系统级退出事件）。

可选增强：

- 允许特定按键中断回放（如 `Esc`）；
- 中断后切回真实输入并记录日志 `replay aborted by user input`。

这样既保证回放稳定，又保留紧急接管能力。

### 3.7 时间基准与确定性策略

回放调度采用“时间戳对齐”而非“帧编号对齐”：

- 每帧累加 elapsed；
- 注入所有 `event.t_ms <= elapsed_ms` 的事件；
- 若某帧跨度较大，允许一次注入多条事件。

`dt` 策略提供两种模式：

- 固定 `dt`（例如 16ms）：更稳定、可重复性更好，适合回归验证；
- 实时 `dt`：更贴近真实运行，但受机器性能波动影响。

首期建议：回放默认固定 `dt`，并允许配置切换实时模式。

确定性边界说明：

- `vn-runtime` 在相同输入序列下应保持脚本行为确定；
- 渲染帧率可变不作为失败判据；
- 音频时序偏差可接受，不纳入首期一致性目标。

### 3.8 坐标与窗口尺寸差异

问题：录制时窗口尺寸与回放时可能不同，绝对像素坐标可能命中错误区域。

首期策略（务实可落地）：

- 录制时保存逻辑尺寸元数据；
- 回放时按比例映射：
  - `x' = x * replay_width / record_width`
  - `y' = y * replay_height / record_height`
- 若尺寸差异超过阈值（如宽高任一方向偏差 > 30%），打印警告日志。

后续可演进策略（非本期）：

- 以 UI 逻辑坐标空间或命名控件 ID 做命中重定向；
- 在录制中加入场景标识与节点校验，防止“脚本分支不同但仍注入点击”。

### 3.9 错误处理与可观测性

外部不受信路径（文件读写/解析）返回 `Result` 并附上下文：

- `ReplayLoadError`（文件不存在、JSON 解析失败、版本不兼容）
- `RecordWriteError`（路径不可写、磁盘失败）

内部不变量（如回放游标越界）可使用 `debug_assert!`/`unreachable!` 保护。

日志建议：

- 录制开始/结束、事件总数、输出路径；
- 回放开始/结束、总事件数、已注入数；
- 回放中断原因；
- 时间漂移统计（可选）。

### 3.10 配置与 CLI 集成

`DebugConfig` 新增字段：

- `record_input: Option<String>`
- `replay_input: Option<String>`

命令行参数可映射到同名配置覆盖（若已有 CLI 到 `AppConfig` 的映射机制则复用）。

优先级建议：

1. CLI 显式参数；
2. 配置文件；
3. 默认值（关闭）。

## 4. 影响范围

| 模块 | 改动 | 风险 |
|------|------|------|
| `host/src/input/mod.rs` | `InputManager` 增加 recorder/replayer 集成与统一注入入口 | 中（输入主路径，需回归） |
| `host/src/input/recording.rs`（新增） | 定义 `InputEvent`、`InputRecorder`、`InputReplayer` 与序列化逻辑 | 中（新子系统，需测试覆盖） |
| `host/src/app/mod.rs` | 启动期按配置初始化录制/回放并注入 `AppState`/`CoreSystems` | 低-中（初始化流程改动） |
| `host/src/config` 相关 | `DebugConfig` 新字段与反序列化兼容 | 低（配置扩展，默认关闭） |
| `vn-runtime` 边界消费 | 不改语义，仅验证回放输入下行为一致 | 低（接口不变） |
| 文档 | 调试手册/RFC 索引更新 | 低 |

## 5. 迁移计划

分阶段实施：

1. **阶段 A：数据结构与文件读写**
   - 引入 `InputEvent` 与 `.jsonl` 编解码；
   - 实现 `InputRecorder`/`InputReplayer` 的独立单元测试。
2. **阶段 B：InputManager 接入**
   - 将真实输入处理提取为统一内部入口；
   - 录制挂在该入口后；
   - 回放按时间戳注入同入口。
3. **阶段 C：配置与启动集成**
   - `DebugConfig` 新增字段；
   - `AppState` 启动时根据配置启用；
   - 明确录制/回放互斥策略。
4. **阶段 D：回归与文档**
   - 增加关键集成测试（等待态、点击推进、选择分支）；
   - 补充使用说明（如何录制、如何回放、如何判读日志）。

兼容性：

- 新字段均为 `Option`，旧配置文件无需修改即可启动；
- 无需迁移现有脚本资源；
- 录制文件建议包含版本号，后续格式演进可向后兼容解析。

## 6. 验收标准

- [ ] 在开启 `debug.record_input` 时，程序可输出合法 `.jsonl`，且包含时间戳与语义事件字段。
- [ ] 在开启 `debug.replay_input` 时，程序可在无人工输入下驱动脚本推进至预期状态（以日志或状态断言验证）。
- [ ] 回放模式下默认忽略真实输入，不影响回放顺序；可保留退出/中断能力。
- [ ] 同一录制文件在同配置下重复回放，`vn-runtime` 的关键状态转移序列一致。
- [ ] 当窗口尺寸与录制尺寸不一致时，系统执行坐标比例映射并输出可观测警告（超阈值时）。
- [ ] 文件解析/写入失败时提供结构化错误与上下文，不因单次失败导致应用异常崩溃。
- [ ] 为新模块与关键输入路径补充测试（单元 + 必要集成），并通过 `cargo check-all`。
- [ ] 文档更新包含：RFC 索引项、调试操作说明、已知限制（渲染/音频非严格确定）。
