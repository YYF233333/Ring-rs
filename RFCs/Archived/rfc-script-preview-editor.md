## 背景

### 原始痛点

当前的叙事开发流程是：编剧编写 `.md` 脚本 → 启动引擎 → 手动导航到对应场景 → 观察效果。反馈循环慢，特别是修改位于脚本后段的内容时。

### 历史教训

**v1**：尝试独立实现预览渲染路径，绕过正常初始化和执行管线，导致大量状态未正确构建（缺背景、对话框、manifest）。

**v2**：提出"预览 = 正常游戏 + 自动驾驶 + 回退"，试图在启动时通过自动输入快进到指定位置。两次实现均存在 bug，问题在于快进路径需要处理大量边界情况（信号等待、UI 结果、动画跳过等），与正常执行路径的耦合度高，维护成本不合理。

### v3 核心理念

> **不做专用预览模式，将预览所需能力拆解为通用引擎特性，集成到正常游戏中。**

预览能力本质上可拆解为三个独立特性：

| 能力 | 本质 | 实现方式 |
|------|------|----------|
| 跳转到指定位置 | 跳过前面的内容 | ~~自动驾驶快进~~ → **存档功能**代替 |
| 单步/任意步回退 | 状态快照恢复 | Snapshot 系统 + 可点击历史记录（集成到正常游戏） |
| 脚本热重载 | 修改后快速看到效果 | Recording 回放 + 热键重启 |

每个能力独立实现、独立可用，不需要专用的 `--preview` 启动模式。

---

## 目标与非目标

### 目标

- 在正常游戏中支持 Backspace 键单步回退到上一个停止点
- 历史记录界面中每条记录可点击，跳转到该处继续游玩
- 历史快照条数不设上限，但不持久化到存档（重新加载后历史为空）
- 热键（F5）触发脚本热重载：重启游戏 + 使用 recording 输入快速推进到刷新前位置
- 跳转到指定位置的需求通过存档功能满足（手动存档到目标位置附近）

### 非目标

- 不做专用预览模式或 `--preview` 启动参数
- 不做启动时自动快进到指定行号（v2 方案已废弃）
- 不做文件系统监听（不用 `notify` crate）
- 不做内置文本编辑器或双向编辑
- 不做 VS Code 扩展集成

## 方案设计

### 能力 1：状态快照与回退（Snapshot 系统）

状态快照是引擎的通用能力，集成到正常游戏流程中。每个停止点自动保存快照，支持单步回退和任意位置跳转。

**快照内容**：

```rust
struct StateSnapshot {
    // Runtime 层（vn-runtime）
    runtime_state: RuntimeState,     // 位置、变量、等待状态、调用栈

    // Host 层
    render_state: RenderState,       // 当前渲染状态（背景、角色、对话等）
    audio_state: AudioState,         // BGM 等音频状态
}
```

**快照策略**：
- 每个停止点（WaitForClick / WaitForChoice）自动保存快照
- 快照条数不设上限（内存中保留本次游玩的完整历史）
- **不持久化到存档**——存档加载后快照历史为空，仅记录加载后的操作
- 快照只保留逻辑状态（路径引用），纹理等资源按需从缓存重建

**回退操作**：
1. 从快照栈中取出目标快照
2. 恢复 RuntimeState（位置、变量、等待状态）
3. 恢复 RenderState（背景、角色、对话等）
4. 恢复音频状态（重新播放对应 BGM 等）
5. 重新加载当前 RenderState 引用的资源纹理

**与 History 的关系**：
- `History`（历史记录）是展示层——用户在历史界面看到的对话列表
- `StateSnapshot` 是状态层——每条历史记录关联一个快照
- 历史记录界面中的每条记录可点击，点击后恢复到对应快照
- History 条数与快照条数同步，同样不设上限、不持久化到存档

### 能力 1 的用户交互

**单步回退**：
- 绑定 **Backspace** 键
- 按下后恢复到上一个停止点的快照
- 可连续按 Backspace 逐步回退

**历史跳转**：
- 打开历史记录界面（现有功能）
- 每条历史记录旁显示可点击区域
- 点击后恢复到该条记录对应的快照，从该处继续游玩

```
┌──────────────────────────────────┐
│         历史记录界面              │
├──────────────────────────────────┤
│  [►] 莉莉丝：「欢迎来到……」      │
│  [►] 旁白：天空渐渐暗了下来       │
│  [►] 莉莉丝：「你准备好了吗？」    │  ← 点击 [►] 跳转到此处
│  [►] 【选择】走左边的路           │
│  [►] 旁白：你踏上了左边的小路      │  ← 当前位置
└──────────────────────────────────┘
```

### 能力 2：脚本热重载（Recording 驱动）

脚本热重载不依赖文件系统监听，而是利用 recording 系统实现。

**流程**：

```
用户修改脚本并保存（在外部编辑器中）
    ↓
用户按下 F5（热重载热键）
    ↓
引擎记录当前 recording 状态
    ↓
引擎重启（重新初始化 AppState、重新加载脚本）
    ↓
使用 recording 中的输入自动快进到重启前的位置
    ↓
到达目标位置后切换到正常交互模式
```

**关键设计**：
- **无需文件监听**：用户主动按 F5 触发，避免 `notify` 依赖和文件监听的复杂性
- **Recording 驱动快进**：recording 已记录了用户从游戏开始到当前位置的所有输入（点击、选择等），重启后 replay 这些输入即可到达相同位置
- **快进期间跳过渲染和等待**：与 headless 模式类似，紧密循环执行，毫秒级完成
- **脚本变更容错**：如果脚本修改导致 recording 输入与新脚本不匹配（如选项数量变化），快进中止在最后一个能匹配的位置，并提示用户

**信号等待的即时完成**（快进期间）：

| 信号 | 快进行为 |
|------|----------|
| `scene_transition` | 立即完成过渡动画，设置最终状态 |
| `scene_effect` | 立即完成效果动画 |
| `title_card` | 立即清除 title card |
| `cutscene` | 跳过视频播放 |

这些通过调用现有的 `skip_all_active_effects()` 并直接发送对应信号来实现。

### 跳转到指定位置（存档方案）

v2 尝试通过自动驾驶快进到指定行号，两次实现都有 bug。该需求改用存档功能替代：

- 开发者在目标位置附近手动存档
- 需要预览时加载该存档，从存档点继续游玩
- 配合 Backspace 回退和 F5 热重载，可以高效迭代

这虽然不如"输入行号直接跳转"自动化，但避免了自动驾驶快进的大量边界问题，可靠性远高于 v1/v2 方案。

## VNRuntime API 扩展

为支持快照/恢复，VNRuntime 需要暴露：

```rust
impl VNRuntime {
    /// 获取当前状态的可克隆快照
    pub fn snapshot_state(&self) -> RuntimeState { ... }

    /// 从快照恢复状态
    pub fn restore_state(&mut self, state: RuntimeState) { ... }
}
```

注：`RuntimeState` 已经 `Clone`，主要需要新增 `restore` 方法。

## 风险

| 风险 | 缓解 |
|------|------|
| 快照内存占用随游玩时间增长 | 快照只保留逻辑状态（路径引用），纹理按需重建；如有必要可后期加 LRU 淘汰 |
| 回退后继续前进时状态不一致 | 回退时完整恢复 Runtime + Render + Audio 三层状态 |
| Recording 与修改后的脚本不匹配 | 快进中止在最后匹配位置，提示用户手动继续 |
| F5 重启的用户体验（闪烁/延迟） | 快进期间不渲染，仅在到达目标后显示画面；可加 loading 遮罩 |
