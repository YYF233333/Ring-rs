# RFC: 脚本预览编辑器

## 元信息

- 编号：RFC-028
- 状态：Proposed
- 作者：claude-4.6-opus
- 日期：2026-03-24
- 影响范围：`host`（新增预览模式或独立工具）、`vn-runtime`（可能需要 API 扩展）
- 前置：无（与 Hub 架构正交，可独立推进）
- 父 RFC：RFC-024（VN+ Hub 架构愿景）
- 灵感来源：Artemis Engine 的脚本预览编辑功能

---

## 背景

当前的叙事开发流程是：编剧编写 `.md` 脚本 → 启动引擎 → 手动导航到对应场景 → 观察效果。这个反馈循环存在以下痛点：

1. **启动成本高**：每次修改后需要重启或手动跳转，特别是修改位于脚本中后段的内容时
2. **缺乏即时预览**：无法在编辑的同时看到渲染效果
3. **定位困难**：编剧想查看特定段落的效果，必须从头播放或手动设置存档点

Artemis Engine 提供了专用的脚本预览编辑器，编剧可以在编辑器中点击脚本的任意位置，立即预览该处的视觉效果。这大幅提升了叙事开发效率。

Ring Engine 已具备实现类似功能的基础：

- `headless` 模式可以无窗口运行 VN 逻辑
- vn-runtime 的确定性执行保证了任意位置的状态可重建
- `ScriptPosition` 和 source map 提供了脚本 ↔ 执行位置的映射

**实施时机：可立即开始设计，不依赖 Hub 架构。直接服务当前纯 VN 开发。**

---

## 目标与非目标

### 目标

- 编剧可以在外部文本编辑器（VS Code 等）编辑 `.md` 脚本，预览窗口实时显示效果
- 支持"跳转到指定行"：点击/指定脚本的某一行，预览器快速渲染该行处的视觉状态
- 文件监听：脚本文件保存后自动刷新预览
- 复用现有渲染管线，不重新实现渲染逻辑

### 非目标

- 不做内置文本编辑器——编剧使用自己熟悉的编辑器，预览器只负责渲染
- 不做双向编辑——不从预览器的可视化操作生成脚本代码
- 不替代正常的游戏运行模式——预览器是开发工具，不是发布产物
- 不支持非 VN 模态的预览（那是各 mode 自己的工具需求）

## 方案草案

### 架构选项

#### 方案 A：引擎内置预览模式

在现有 host 中增加一个启动模式（如 `--preview` flag），进入预览模式而非正常游戏模式。

```
host --preview --script chapter1.md --line 42
```

预览模式下：
- 左侧/底部显示脚本文本（只读，egui 文本区域）
- 右侧/主区域显示渲染效果
- 监听脚本文件变更，自动刷新
- 可通过 UI 控件或快捷键跳转到指定行

**优势**：完全复用现有渲染管线，无额外集成成本。
**劣势**：与游戏运行模式共用二进制，启动参数多。

#### 方案 B：独立预览工具

新增一个 workspace member（如 `tools/preview`），编译为独立可执行文件。

**优势**：关注点分离，不增加 host 复杂度。
**劣势**：需要复用 host 的渲染管线，可能导致代码重复或需要提取渲染库。

#### 方案 C：VS Code 扩展 + LSP

编写 VS Code 扩展，在编辑器侧栏中嵌入预览面板，通过 LSP 或自定义协议与引擎通信。

**优势**：编辑和预览在同一窗口，体验最佳。
**劣势**：开发成本最高，需要 VS Code 扩展开发知识。

### 初步倾向：方案 A

方案 A 的成本最低，且能验证核心价值（即时预览是否真的提升效率）。如果价值验证通过，可以在后续迭代中升级为方案 C。

### 核心技术挑战："跳转到指定行"

预览器的核心能力是"给定脚本的第 N 行，渲染出该行处的视觉状态"。这需要：

1. **状态重建**：从脚本开头快速执行到第 N 行，收集沿途的所有 Command，重建渲染状态
2. **跳过等待**：执行过程中自动跳过所有 `WaitingReason`（不需要用户交互）
3. **处理分支**：遇到 `choice` 时需要默认选择策略（选第一项？或允许用户预设选择路径？）
4. **增量更新**：编辑脚本后，只需从变更点之前的最近状态快照重新执行，而非从头

vn-runtime 的确定性执行模型天然支持 1 和 2。3 需要一个"自动选择"策略。4 是性能优化，可以在第一版中省略（对中等长度的脚本，从头执行也很快）。

### 文件监听

```
启动 → 加载脚本 → 执行到指定行 → 渲染
  ↑                                  │
  └──── 文件变更检测 ←────────────────┘
```

使用 `notify` crate 监听 `.md` 文件变更。变更后重新解析脚本，重新执行到当前行。

## 实施路径

### Phase A：最小可用预览器

1. 在 host 中添加 `--preview` 启动模式
2. 实现"指定脚本 + 行号 → 快速执行 → 渲染状态"的核心链路
3. 使用 `notify` 监听文件变更，自动刷新
4. 简单 egui UI：显示当前行号、前进/后退按钮

### Phase B：交互增强

1. 脚本文本面板（只读，可点击行号跳转）
2. 分支路径选择面板（遇到 choice 时选择预览哪条路线）
3. 变量面板（显示当前 RuntimeState.variables）
4. 性能优化：状态快照缓存，增量执行

### Phase C：编辑器集成（可选）

1. VS Code 扩展：在编辑器侧栏嵌入预览
2. 通过 WebSocket 或 stdio 与预览引擎通信
3. 光标位置同步：编辑器光标移动时预览器自动跟随

---

## 风险

| 风险 | 缓解 |
|------|------|
| "跳转到指定行"在复杂脚本中很慢 | 先不优化，验证是否真的有性能问题；必要时加状态快照 |
| 分支路径使预览不确定 | 第一版默认选第一个 choice，后续加路径选择面板 |
| 预览模式与正常模式行为分叉 | 复用同一套 update/render 代码，预览模式只是"自动输入 + 跳过等待" |
| 维护成本 | 方案 A 的增量代码量小（主要是启动模式分支 + 文件监听） |

## 验收标准

### Phase A

- [ ] `host --preview --script <path>` 可启动预览模式
- [ ] 指定行号后可正确渲染该行的视觉状态（背景/角色/对话）
- [ ] 文件保存后自动刷新预览
- [ ] 前进/后退按钮可逐行步进
- [ ] 遇到 choice 时默认选择第一项继续执行
- [ ] `cargo check-all` 通过
