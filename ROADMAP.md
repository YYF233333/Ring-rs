## 下一阶段开发计划（阶段 11）

> 主题：**脚本语法补齐（音频等）** + **立绘资源/布局元数据系统**

### 11.1 脚本语法补齐：音频/控制流 ✅

- **已完成**
  - `<audio src="..."></audio>`：SFX 播放一次
  - `<audio src="..."></audio> loop`：BGM 循环播放
  - `stopBGM`：停止当前 BGM（带淡出）
  - `goto **label**`：无条件跳转
  - 资源路径解析规则调整：素材路径相对于脚本文件，且路径已规范化（`std::path`）
  - BGM 切换自带交叉淡化效果（1秒）
  - 错误处理：文件不存在/格式不支持时打印错误但不崩溃
  - ~~音量/静音~~：由玩家在设置中控制，脚本层不实现
- **验收标准** ✅
  - 脚本可完整覆盖：播放 BGM → 切换 BGM（自动交叉淡化）→ 播放 SFX → stopBGM

### 11.2 资源管理（立绘）：anchor + pre_scale + preset 布局系统 ✅

- **背景**：立绘尺寸/构图不统一，无法用单一缩放规则保证"站位稳定/构图一致"。
- **核心想法**：为**每组立绘**提供可配置的 `anchor`（重心/对齐点）与 `pre_scale`（预处理缩放），再叠加**全局 preset**（点位 scale/偏移），使不同立绘在屏幕上呈现一致的相对效果。

- **已完成**
  - `assets/manifest.json` 资源清单：
    - `characters.groups`: 立绘组配置（anchor + pre_scale）
    - `characters.sprites`: 立绘路径到组的显式映射
    - `presets`: 九宫格站位预设（x, y, scale）
    - `defaults`: 默认配置（未配置立绘的兜底）
  - `host/src/manifest/mod.rs` 模块：
    - Manifest 数据结构定义（serde 序列化/反序列化）
    - `get_group_config()`: 查找顺序 = 显式映射 → 路径推导 → 默认配置
    - `infer_group_id()`: 兜底规则（目录名 / 文件名前缀）
    - 4 个单元测试覆盖核心场景
  - `Renderer::render_characters()` 重写：
    - 基于 anchor + pre_scale + preset 计算位置和缩放
    - 删除旧的硬编码 `position_to_screen_coords()` / `scale_character_size()`

- **验收标准** ✅
  - 元数据缺失时有合理默认值（不崩溃，可用）
  - 新立绘组只需编辑 manifest.json，无需改代码

## 下一阶段开发计划（阶段 12）

> 主题：**架构性改动（先做会影响项目结构的部分）**

### 12.1 存档/读档系统（结构性能力）

- **为什么是“结构性改动”**
  - 需要为 `RuntimeState` 建立**稳定的序列化格式**、**版本兼容策略**与**跨版本迁移**的约束
  - 需要 Host 层提供统一的存储路径、slot 管理、错误处理与 UI 入口
- **工作项**
  - Runtime
    - 定义 SaveData（包含：脚本 id/当前位置、RuntimeState、必要的渲染/音频可恢复信息）
    - 增加 `save_version` 与兼容策略（至少“向后不崩溃，提示不可读/需迁移”）
  - Host
    - 存档文件布局（例如 `saves/slot_001.json`）与读写 API
    - UI/输入：提供最小可用入口（快捷键保存/读取；后续再做 UI 面板）
- **验收标准**
  - 能在脚本模式下：保存 → 退出 → 重新运行 → 读取并恢复到同一段对话/同一背景与角色状态
  - 错误场景（文件不存在/损坏/版本不匹配）不崩溃且提示清晰

### 12.2 “历史记录”数据模型（结构性能力，先做底座）

- **为什么是“结构性改动”**
  - 历史回看功能最终依赖对话/选择/跳转的**事件日志模型**，最好先把结构定下来
- **工作项**
  - Runtime：定义 HistoryEvent（对话/旁白/章节标记/选择结果等），在 tick 时可选记录
  - Host：先不做 UI，仅确保数据可持久化/可回放（与存档系统对齐）
- **验收标准**
  - 任意脚本推进过程能生成结构化历史记录，并可随存档保存/读取

### 12.3 资源与配置治理（结构性能力）

- **目标**：把“路径/配置/资源元数据”统一成可演进的体系，避免后续 feature 越堆越散。
- **工作项**
  - 固化 `assets/manifest.json` 的 schema 与默认值（已完成的基础上补文档/校验）
  - 增加运行配置（debug 开关、默认字体、资源根目录、存档目录等）集中管理
- **验收标准**
  - 关键路径（assets root、manifest、saves）都可配置且有默认值
  - 配置缺失/错误时不崩溃，有明确错误信息
