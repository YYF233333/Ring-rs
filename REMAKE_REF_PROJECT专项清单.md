# ref-project 重制专项清单（体验复刻导向）

> 目标：复刻原作**游戏体验**，而不是复刻 Ren'Py 的实现逻辑。  
> 原则：去除 Ren'Py 历史包袱，使用 Ring-rs 更清爽的架构；只要主观体验等价，允许细微效果差异。

---

## 1. 复刻标准（先定标）

### 1.1 体验等价判定
- [ ] **叙事等价**：剧情顺序、分支结果、章节解锁与结局触发一致。
- [ ] **演出等价**：核心镜头语言一致（转场节奏、角色入退场、黑白场切换、关键 CG 时机）。
- [ ] **交互等价**：玩家操作路径一致（主菜单/设置/历史/存读档/快进自动）。
- [ ] **听感等价**：BGM/SFX/语音（如保留）的触发时机与层次一致。

### 1.2 明确允许差异
- [ ] 不要求 Ren'Py 语法 1:1 兼容。
- [ ] 不要求 transform/transition 内部算法完全一致。
- [ ] 不要求 GUI 像素级一致（保留美术风格与信息层级即可）。

---

## 2. P0（先做：保证“能完整玩且感觉对”）

### P0-1 内容迁移流水线（弱语法依赖，强结果导向）
- [x] 转换所有剧情相关的`rpy`（已搬迁至 `assets/scripts/remake/{summer,winter}`，保留原语义文本）
- [x] 重点保证剧情与事件时序正确，不追求语法逐项映射。（主流程已在 `assets/scripts/remake/main.md` 编排）
- [x] 未实现功能总结（见 `assets/scripts/remake/README.md` 的“未实现特性与实现计划”章节）



### P0-2 演出能力最小闭环
- [ ] 提供一套统一的“转场描述能力”（替代 Ren'Py 的 `tran_*` 家族）。
- [ ] 覆盖原作高频视觉类型：普通溶解、黑场过渡、rule/mask、冲击镜头、模糊进出。
- [ ] 支持“组合演出”编排（例如：黑场 + Pause + 反向开场）。
- [ ] 验收：`prologue`、`3-5`、`3-7`、`ending` 的关键段落观感一致。

### P0-3 文本节奏与窗口控制
- [ ] 支持对话节奏控制能力（等待、续接、加速、打字速度切换）。
- [ ] 支持脚本驱动的窗口显隐策略（用于关键演出段）。
- [ ] 验收：关键台词段落（含停顿与续接）阅读节奏与原作接近。

### P0-4 持久化与章节门控
- [ ] 建立全局持久化域（用于通关状态、章节解锁、菜单分支）。
- [ ] 对齐原作 `complete_summer` 类行为（通关后入口变化）。
- [ ] 验收：首通后重启游戏，菜单与可进入章节状态正确。

### P0-5 核心系统 UI 可用
- [ ] 主菜单、设置、历史、存档、读档、快进/自动/跳过稳定可用。
- [ ] 优先保障信息架构与流程一致，不追求 Ren'Py screen 逐条复刻。
- [ ] 验收：从标题进入任意章节并完成一次存读档闭环无问题。

## 3. P1（增强：提升“像原作”的主观感受）

### P1-1 角色演出语义层
- [ ] 抽象角色入场/退场/换表情/镜头位置语义（不直接照搬 Ren'Py transform 结构）。
- [ ] 构建常用镜头预设（单人、双人、三人站位与近远景）。
- [ ] 验收：随机抽查 4 个章节，角色镜头与原作主观一致。

### P1-2 音频高级体验
- [ ] 支持 BGM 平滑暂停/恢复与跨段衔接策略。
- [ ] 三通道混音（music/sound/voice）与设置页联动。
- [ ] 验收：关键段（如 `prologue`）BGM 的情绪曲线一致。

### P1-3 视频与终章表现
- [ ] 支持过场视频播放、跳过与播放后流程回归。
- [ ] 验收：`ending` 视频段完整可用，跳过不破坏剧情状态。

## 4. P2（优化：长期可维护与生产效率）

### P2-1 复刻检查器（体验视角）
- [ ] 新增专项静态检查：
  - 高风险演出段未映射
  - 文本节奏标签丢失
  - 资源缺失或路径异常
  - 菜单入口与解锁状态异常

### P2-2 内容生产规范
- [ ] 制定重制脚本规范（命名、章节切分、演出标注）。
- [ ] 建立“转换后脚本 + 演出回放样例”的回归样本库。

### P2-3 视觉一致性迭代
- [ ] UI 风格持续贴近原作（按钮、字体、层次、动效节奏）。
- [ ] 但保持当前引擎架构清晰，不回退到 Ren'Py 风格耦合实现。

---

## 5. 非目标（明确不做）

- 不做 Ren'Py 全语法、全 API 兼容层。
- 不复刻开发者工具链行为（控制台、调试快捷键等）。
- 不因“兼容旧实现”破坏当前引擎分层与可维护性。

---

## 6. 建议实施顺序（短期）

1. **先打通可玩版本**：P0-1 ~ P0-5  
2. **再拉齐观感与听感**：P1-1 ~ P1-3  
3. **最后做规模化与长期维护**：P2
